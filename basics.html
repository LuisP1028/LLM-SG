Part I: Basics

1. The Retrieval Pipeline
Core Mechanism

The Flow: Question → Vector → Database Match → Context Extraction → LLM Prompt.

  1. Vectorization: The user input is converted into a list of numbers (vector) capturing semantic meaning.
  2. Similarity Search: The system compares this vector against the database to find the K most similar records.
  3. Augmentation: Matching info is extracted and appended to the prompt, acting as the knowledge base for the answer.
Distance Metrics

How we measure the "gap" between vectors determines what is considered relevant.

  • Manhattan (Dimensional): Treats dimensions independently. Good when criteria are separate. Prevents single-dimension domination.
  • Euclidean (Magnitude): Measures spatial distance. Sensitive to overall magnitude; squaring amplifies large gaps.
  • Cosine (Similarity): Measures directional alignment only. Ignores magnitude/scale. Best for semantic meaning.
2. CDC (Change Data Capture)
The Role of CDC

Links the Data Warehouse (Source) to the Feature Store (Target).

It captures raw data changes (transaction logs) to trigger transformation logic immediately. This updates computed features in real-time, avoiding slow bulk re-processing.

Push vs. Pull Strategies
PUSH (Active): - Source detects change -> Transmits immediately. - Result: Near real-time sync. - Safety: Messaging system acts as a buffer. PULL (Passive): - Source logs changes -> Target periodically requests (polls). - Result: Reduces source load, introduces latency. - Safety: Logs are buffered for safety.
3. CDC Implementation Types
Trigger-Based (Precision)

Mechanism: Pre-defined SQL triggers execute on data change, writing details to a tracking table.

  • Pros: Captures everything (including deletes). Free, built-in, easy to implement.
  • Cons: Slows down every data entry (extra write operations). Best for strict auditing on light workloads.
Log-Based (Performance)

Mechanism: Parses immutable binary transaction logs (WAL) at the storage layer.

  • Pros: The "Gold Standard." Zero load on the main DB. Instant capture.
  • Cons: Complex, expensive, often vendor-specific. Best for mission-critical speed.
Time-Based (Simplicity)

Mechanism: Query for records with a "Last Modified" timestamp newer than the previous sync (High Water Mark).

  • Pros: Easy to build (just a query).
  • Cons: Misses deletions (records vanish before scan). Slows DB while checking. Best for simple periodic updates.
4. Training Parameters
Schedulers

Linear: Drops learning rate at a constant rate. Predictable, steady cooling.

Cosine: Bell-curve decay. Stays high longer, drops fast, ends with a very slow tail. Allows for precise fine-tuning in final steps.

Efficiency & Context

Batch Size & Accumulation: Large batches give smoother gradients but eat VRAM. Gradient Accumulation mimics large batches by summing gradients over small "mini-batches".

Packing (Multipack): Stitches multiple short examples into one sequence to prevent VRAM waste on padding. Uses attention masks to "wall off" samples so the model doesn't confuse contexts.

Optimizers

AdamW: The gold standard. Uses "moments" and weight decay.

  • 32-bit: Highest precision.
  • 8-bit: Quantized. Saves massive VRAM with similar quality.

Paged Optimizers: Offload state to CPU RAM when GPU is full (prevents OOM crashes).
