Part III: RAG Architectures

1. RAG Overview & Modules

[cite: 1] RAG: Pulling new, updated, tailored information from a vector store to concatenate as context to a user prompt, before feeding a foundational model.

[cite: 2] A RAG system is composed of three main modules:

  • Ingestion pipeline: A batch or streaming pipeline used to populate the vector DB
  • Retrieval pipeline: A module that queries the vector DB and retrieves relevant entries to the user’s input
  • Generation pipeline: The layer that uses the retrieved data to augment the prompt and an LLM to generate answers
HOW ARE MODULES CONNECTED OVERVIEW
  1. On the backend side, the ingestion pipeline runs either on a schedule or constantly to populate the vector DB with external data.
  2. [cite: 3] On the client side, the user asks a question.
  3. [cite: 4] The question is passed to the retrieval module, which preprocesses the user’s input and queries the vector DB.
  4. [cite: 5] The generation pipelines use a prompt template, user input, and retrieved context to create the prompt.
  5. [cite: 6] The prompt is passed to an LLM to generate the answer.
  6. The answer is shown to the user. [cite: 7]
6. Vanilla RAG Optimization
EMBEDDING FINE TUNING SIMPLIFIED: * Vectorization: The model maps the Query and Candidates (passages) to coordinates in embedding space. * Scoring: It computes similarity scores (e.g., dot product) between the Query and every Candidate in the current batch. * Alignment: The loss identifies the Positive (correct match). It maximizes the Query-Positive score, mathematically pulling their vectors closer together. * Differentiation: The loss function calculates a penalty if the model incorrectly places the Query near wrong answers (Negatives). This forces the model to recognize and repel distractors. * Update: The system uses that penalty to tweak the model's weights. This permanently adjusts the mapping logic so the Query lands closer to the Positive and further from Negatives next time.
