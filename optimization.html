1. Inference Optimization & Decoding
INFERENCE OPTIMIZATION TECHNIQUES

Optimization focuses on three critical metrics: latency (time to first token), throughput (tokens per second), and memory footprint. Standard deployment often underutilizes hardware. To achieve 2‚Äì4X efficiency gains, developers employ:

  • Model Parallelism: Distributing the model‚Äôs workload across multiple processing units.
  • Weight Quantization: Compressing model weights to lower precision to save memory.
  • Speculative Decoding: Using smaller models to "guess" tokens, speeding up the main generation.

Choosing the right technique depends on your specific bottleneck:

  • KV Cache: Non-negotiable for all LLM deployments to avoid ùëÇ(ùëõ¬≤) re-computation.
  • Static Caching (torch.compile): Best for latency-critical apps with fixed sequence limits; use when you need maximum GPU utilization without recompilation overhead.
  • Continuous Batching : Best for high-throughput APIs; use when serving multiple concurrent users to prevent "stragglers" from stalling the GPU.
  • Tensor Parallelism: Best for large models (e.g., 70B+) where memory bandwidth is the bottleneck; use if you have a compatible, smaller draft model.
  • Speculative Decoding : Best for tasks with high input-to-output overlap (e.g., summarization, document Q&A, editing); use when you want the acceleration of speculative decoding without the VRAM cost or complexity of managing a second draft model.
  • Quantization: Best for memory-constrained hardware; use when the model won't fit in VRAM or to increase batch sizes.
ENCODER ONLY INFERENCE
  1. Tokenizing the input prompt and passing it through an embedding layer and positional encoding.
  2. Computing key and value pairs for each input token using the multi-head attention mechanism.
  3. Generating output tokens sequentially, one at a time, using the computed keys and values.

First, the prefill phase (Steps 1‚Äì2) processes the input prompt in parallel, which GPUs handle efficiently. Second, the decoding phase (Step 3) generates tokens sequentially. Because each new token depends on all previous ones, this step cannot be parallelized, leaving hardware underutilized and creating the primary bottleneck for generation speed.

KEY-VALUE CACHING

When the model predicts the 101st token, the Keys (ùêæ) and Values (ùëâ) for tokens 1‚Äì100 are identical to what they were during the previous step. KV Caching exploits this by saving those vectors in GPU memory.

  1. The Retrieval: Instead of re-processing the entire sequence, the model only generates a Query (ùëÑ), Key (ùêæ), and Value (ùëâ) for the newest token.
  2. The Lookup: The new ùëÑ scans the cached ùêæs (the "addresses") of all previous tokens to determine relevance.
  3. The Extraction: It then pulls the corresponding information from the cached ùëâs (the "payloads").

By only calculating one new ùêæ/ùëâ pair per step and reading the rest from the cache, the model avoids redundant math, transforming an ùëÇ(ùëõ¬≤) computational burden into a much faster linear process.

TORCH.COMPILE ‚Äî STATIC CACHING

Standard PyTorch runs line-by-line, creating communication lag between the CPU and GPU. torch.compile "fuses" these separate steps into a single optimized kernel. Instead of sending many small tasks, it sends one "super-instruction," letting the GPU work at max speed without interruptions.

In a computer's memory, a tensor is defined by its shape. When the KV cache is dynamic, the tensor representing it looks like this:

Step 1: [1, 1, hidden_dim] Step 2: [2, 1, hidden_dim] Step 3: [3, 1, hidden_dim]

Every time that first number (the sequence length) increases, the GPU sees a "new" object. It has to "re-compile" or look up a new way to handle a 3-row matrix versus a 2-row matrix, the GPU cannot reuse the exact same machine-code instructions it just used.

Static Caching sets the dimension to a maximum (e.g., [2048, 1, hidden_dim]) immediately. Because the shape never changes, the GPU can use one single, hyper-optimized execution path for every single step of the generation.

CONTINUOUS BATCHING

If processing batch A, and 80% finishes, with 20% remaining; In continuous batching, the 80% that finishes isn't "removed" in a traditional sense‚Äîit is evicted and immediately replaced.

Instead of waiting for that final 20% (the "straggler") to finish, the system inserts new requests into the available slots of the current batch. This ensures the GPU never sits idle while waiting for a single long-running task to complete.

SPECULATIVE DECODING

In standard generation, a massive model must load all its parameters from memory just to produce one token. The bottleneck isn't the math; it's the delivery. Loading 70B parameters to calculate one token wastes 99% of the GPU's computing power.

  1. To fix this, a Draft Model (small and fast) "guesses" a sequence of future tokens (e.g., 5 tokens). A small draft model has fewer layers and parameters, so it fits better in memory and generates 5 tokens fast.
  2. The large model validates the draft‚Äôs ‚Äúfuture‚Äù tokens by running them through its own layers as a batch. If the draft guessed [ùê¥, ùêµ, ùê∂, ùê∑, ùê∏] and the large model agrees with all five, you‚Äôve effectively skipped 5 expensive memory-loading cycles.
  3. If the large model disagrees at ùê∂, it keeps ùê¥ and ùêµ, provides its own ‚Äúcorrect‚Äù ùê∂, and discards the rest. You still come out ahead because you got 3 tokens for the price of 1.

Both models must use the same tokenizer ‚Äî Models use IDs, not words. If ID 1520 is "apple" to the draft model but "cat" to the large model, the math breaks. They need a shared "dictionary" to ensure ID numbers represent the same concepts.

PROMPT LOOKUP DECODING

Prompt Lookup Decoding accelerates text generation by exploiting the high probability that the output will repeat phrases found in the input (e.g., in summarization). It removes the need for a separate "draft model" used in standard speculative decoding.

Here is the step-by-step process:

  • Generate: The main Large Language Model (LLM) produces a few current tokens (words) as normal.
  • Match: The system scans the original input prompt to see if this exact sequence of tokens appears somewhere in the source text.
  • Speculate: If a match is found, the system "copies" the subsequent tokens directly from the prompt, hypothesizing that the model will continue to follow that source text.
  • Validate: The LLM checks all these speculated tokens simultaneously in a single parallel pass. If they are correct, they are accepted instantly; if not, the model discards them and generates the next token normally.
SELF-ATTENTION DEFINITIONS

In Transformers, Self-Attention is the mechanism that allows the model to understand context by relating words to one another. This process relies on three vectors generated for every token:

  • Query (ùëÑ): This is used to assess context between the token and the surrounding tokens.
  • Key (ùêæ): A feature vector describing the cluster each token‚Äôs position in a high dimensional vector space.
  • Value (ùëâ): Contains the actual features of each token.
2. Model Parallelism
Data Parallelism (DP)

DP replicates the entire model onto every GPU, allowing each to process a different data subset simultaneously. DP is superior for throughput because it minimizes communication; GPUs only sync once per step (after gradients).

When to choose: Use Data Parallelism when your bottleneck is data volume, not model size. It is the easiest to implement and scales linearly across nodes, provided your model isn't so large that it leaves no room for the data batch itself.

  • Mechanism: In training, GPUs synchronize by averaging gradients to maintain identical weights. In inference, DP facilitates concurrent processing, where multiple requests are handled in parallel.
  • The Bottleneck: Because every GPU must store the full model, DP is limited by the memory of a single device.
  • Trade-off: It significantly boosts throughput, but involves high communication overhead and redundant memory usage, making it inefficient for models that exceed individual GPU capacity.
Pipeline Parallelism (PP)

Pipeline Parallelism partitions a model vertically by layers. Instead of every GPU holding the entire model, the layers are divided into segments and distributed across a sequence of GPUs (the "degree of parallelism").

When to choose: Use PP when memory is the bottleneck. It allows you to "stack" GPUs to hold the model. It is ideal for scaling between different servers because it only transmits activations, which is less data-intensive than TP.

  • The Assembly Line: Data moves through the system like a factory conveyor belt. GPU 1 processes the first segment and passes the activations (intermediate outputs) to GPU 2. In training, this process reverses during the backward pass, with gradients flowing back down the chain.
  • Memory Efficiency: This is the primary driver for PP. Because each device only stores a fraction of the model‚Äôs weights and the corresponding optimizer states, you can run models that are far too large for any single GPU's VRAM.
  • The "Pipeline Bubble": The main drawback is the sequential dependency. GPU 4 cannot start its work until GPUs 1, 2, and 3 have finished theirs. This creates "bubbles"‚Äîperiods of idle time where hardware sits unproductive while waiting for data from the preceding stage.
  • MITIGATING PIPELINE BUBBLES: Instead of waiting for a massive batch to clear Stage 1, the first micro-batch is passed to Stage 2 immediately.
Tensor Parallelism (TP)

Tensor Parallelism splits a model horizontally by partitioning individual weight matrices (tensors) across multiple GPUs.

When to choose: Use TP for latency-sensitive tasks and ultra-large models within a single server. It requires NVLink speeds to handle constant syncing. It is often combined with PP (forming 3D parallelism) to manage both layer depth and layer width.

  • The Mechanism: Instead of layers being on different devices (PP), a single layer‚Äôs math is shared. Large operations, like MLP weights or Attention Heads, are sliced into shards. GPUs receive the same input, compute their specific shard's result, and use an All-Reduce operation to sync and combine partial results into a final output.
  • Granularity: It is highly efficient for Self-Attention since heads are naturally independent. However, layers like LayerNorm or Dropout are often replicated or split via Sequence Parallelism (by input length) because they require a global view of the data.
  • The Constraint: TP requires extremely high-speed interconnects (like NVLink) because GPUs must communicate multiple times within every single layer, making it best suited for GPUs within the same server.
COMBINING MODEL PARALLELISM APPROACH

The three methods are orthogonal, meaning they can be stacked like building blocks to form 3D Parallelism.

  • The Strategy: You split a model into Pipeline stages to fit across servers, use Tensor parallelism within each server to accelerate wide layers, and then wrap the whole setup in Data parallelism to process multiple batches at once.
  • The Goal: This hybridization lets you balance memory (PP), latency (TP), and throughput (DP). It is the industry standard for training "frontier" models (like GPT-4 or Llama-3-70B+) that are too massive for any single technique to handle efficiently.
3. Model Quantization
MODEL QUANTIZATION

Quantization refers to the process of representing the weights and activations of a neural network using lower-precision data types.

  • FP32 (Full): Highest precision; uses 32 bits per number. Accurate but slow and memory-intensive.
  • BF16/FP16 (Half): 16-bit formats. BF16 is preferred for LLMs because it preserves a larger range (exponent), preventing mathematical "overflow" during training.
  • INT8 (Quantization): An 8-bit integer format. Using techniques like absmax, you can compress high-precision weights into 8-bit buckets, drastically reducing VRAM usage at the cost of some "granularity" in the numbers.
Quantization Methods

Post-Training Quantization (PTQ): PTQ is a straightforward technique where the weights of a pre-trained model are directly converted to a lower precision format without any retraining.

Quantization-Aware Training (QAT): QAT performs quantization during the training or fine-tuning stage, allowing the model to adapt to the lower precision weights.

Absmax Quantization

This is a symmetric scaling method. You find the absolute maximum value in a tensor and use it to map everything into the [-127, 127] range.

The Problem: It assumes the data is centered around zero. If your weights are skewed (e.g., all positive), you waste half of your available "buckets," leading to higher rounding errors.

Zero-Point Quantization

Handles data that isn‚Äôt centered at zero. If your weights range from 2.0 to 10.0, Absmax wastes half the ‚Äúslots‚Äù because it forces a symmetric range (e.g., ‚àí10 to 10) even when negative values don‚Äôt exist.

The Goal: Map your actual minimum (e.g., 2.0) to ‚àí128 and your actual maximum (e.g., 10.0) to 127. This ensures the entire 8-bit ‚Äúbudget‚Äù is used for your specific data.

The Zero-Point (ùëç): Since your data is shifted (it doesn‚Äôt start at 0), you calculate an offset. The zero-point is the specific integer in the 8-bit range that represents the original 0.0.

THE OUTLIER PROBLEM

Because quantization scales the entire range based on extreme values. If most weights are approx 0.1 but one is 10.0, the mapping forces 99.9% of the weights into a tiny fraction of the available integer "slots" (e.g., just 0 and 1). This "squishing" destroys the model's nuance.

Decomposing the matrix into two separate pieces before the math happens. Think of it like a filtered bypass. When the model encounters a matrix, it doesn't process it as one block. It scans for values exceeding a threshold (like 6.0).

  1. The Extraction: It literally pulls those outlier columns out and places them into a separate ‚ÄúHigh-Precision‚Äù matrix (ùê¥).
  2. The Remainder: What‚Äôs left is a matrix of ‚Äúnormal‚Äù values. Because the outliers are gone, this matrix can now be quantized to ùêµ with high resolution.
  3. Parallel Math: The computer performs two separate multiplications: the ‚Äúskinny‚Äù ùê¥ lane and the ‚Äúbulk‚Äù ùêµ lane.
  4. The Merge: The results (now both in ùê¥) are summed back together.

The COMPUTATIONAL SAVINGS isn‚Äôt in the final addition; it‚Äôs in the matrix multiplication, which is the heaviest part of the workload.

OPTIMIZED ATTENTION

Standard attention scales quadratically, causing the KV cache‚Äîthe memory storing temporary token data‚Äîto overflow during long sequences.

PagedAttention solves this by partitioning the KV cache into small, fixed-size blocks that can be scattered across physical memory rather than requiring one continuous slab.

Key Advantages: Near-Zero Waste: Dynamic allocation allows for much higher batch sizes, increasing throughput.

  • HBM (High Bandwidth Memory): Think of HBM as the GPU's "warehouse." It is massive (e.g., 80GB on an H100) and stores all your model weights and long-term data. However, moving data from HBM to the processor is slow.
  • SRAM (Static Random-Access Memory): SRAM is the "workbench" located right on the GPU chip. It is incredibly fast‚Äîorders of magnitude faster than HBM‚Äîbut it is tiny (often only tens of megabytes).
SOFTMAX

When the GPU compares a Query (Q) to all Keys (K), it gets a row of raw similarity scores. Some are high, some are low, and some are negative. These raw numbers are hard for a computer to use as "weights."

The Role of Softmax:

  • Normalization: It turns those raw scores into probabilities that sum up to 1.0 (or 100%).
  • Highlighting: It uses an exponential function (e^x), which aggressively "widens the gap." A slightly higher raw score becomes a much higher probability, while low scores are crushed toward zero.

Online Softmax: If you find a new, larger maximum (max_new), your old results (e^(x - old_max)) are now wrong. To fix them without re-reading the raw data from the ‚Äúwarehouse‚Äù (HBM), you multiply the old results by e^(old_max - max_new). Mathematically, this "shifts" the old values to the new baseline, and allows it to compute mathematically perfect softmax values incrementally, without ever storing the full ùëÅ √ó ùëÅ matrix.

THE HOW (FLASHATTENTION)
  1. The Tiling Strategy: The process begins by partitioning the large Query (ùëÑ), Key (ùêæ), and Value (ùëâ) matrices into small, manageable blocks (tiles). These tiles are sized specifically to fit into the SRAM, the high-speed ‚Äúworkbench‚Äù right next to the GPU cores. You can visualize this like a kernel in a CNN.
  2. Loading the Workbench: The GPU pulls one block of ùëÑ and one block of ùêæ from the HBM into the SRAM. Because these blocks are small, this transfer is efficient. Once they are on the workbench, the GPU performs the matrix multiplication (ùëÑ √ó ùêæ) at lightning speed.
  3. The Online Softmax Trick: As it processes each block, it keeps track of two local variables: the running maximum and the running sum of exponentials. When a new block is processed, the GPU uses these variables to "rescale" the previous weights.
  4. Computing the Weighted Sum: Once the partial softmax for the current block is ready, the GPU pulls the corresponding block of ùëâ into SRAM. It multiplies the partial scores by these values to get a partial output. This preserves information with high scores and effectively mutes the rest, adding this "weighted" data to the final output.
  5. The Final Write-Back: The final output is written to HBM chunk-by-chunk as each row finishes, not all at once.
